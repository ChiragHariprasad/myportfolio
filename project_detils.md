# 01 — GENESIS

### Generative Engine for Networked, Embedded, Spatial Infrastructure Synthesis

**Domain:** AI-Driven Urban Planning & Layout Synthesis  
**Status:** Completed | Patent Published on 27.03.2026

---

### Overview

GENESIS is a rule-aware generative system that automates urban layout planning from scratch. Given only a site boundary and basic parameters, it outputs a fully code-compliant city layout – complete with road networks, city blocks, and subdivided plots. It enforces Indian National Building Code (NBC) regulations directly inside the generation loop, runs fast physics-proxy simulations for traffic, drainage, and wind, and scores each generated layout using a multi-domain KPI framework. It was designed to replace the slow, manual, iteration-heavy process of urban design with an automated, objective, and reproducible pipeline.

---

### Key Features

**A – Automated Boundary-to-Layout Generation**  
Converts raw site boundaries and basic parameters into complete urban layouts: road networks, blocks, and individual plots. Multiple candidate layouts are generated per run.

**B – Regulation-by-Design (Fail-Fast Compliance Engine)**  
Constraints are enforced inside the generation loop, not after. Infeasible layouts are rejected early and regenerated automatically. Enforced constraints include road width, plot limits, FAR (Floor Area Ratio), setbacks, and land-use rules.

**C – Integrated Multi-Domain Performance Evaluation**  
Three fast proxy simulations run on every generated layout:  
- Traffic: graph-based routing, congestion penalties, Level of Service (LOS) metrics  
- Drainage: D8-style flow accumulation, flood-risk penalty scoring  
- Wind/Ventilation: Gaussian wake-based ventilation corridor scoring  

**D – KPI-Based Scoring and Ranking Engine**  
All simulation outputs are normalized into KPIs, aggregated into a composite score (0–100), and used to rank layouts. A Top-K selector identifies the best alternatives.

**E – Interoperable Export Pipeline**  
Outputs are packaged in three standard formats: JSON (structured data contract), OBJ/MTL (3D visualization), and SUMO-ready XML (traffic simulation integration).

**F – End-to-End Prototype UI**  
Streamlit-based UI with interactive map-based boundary drawing (Folium + Leaflet), layout generation controls, side-by-side comparison, and download/export interface.

### Algorithm Stack

**A – Generative Layout Synthesis**  
Procedural generation pipeline: site boundary → road network generation → block formation → plot subdivision. Multiple candidate layouts are produced per run via parameterized variation.

**B – Rule Engine (Compliance-by-Design)**  
Deterministic constraint enforcement logic: `Generate → Validate → Fail? Regenerate → Pass? Simulate + Score`  
Constraints checked: road widths, FAR, setbacks, plot limits, land-use compatibility.

**C – Traffic Proxy Simulation**  
Graph-based shortest path assignment over the generated road network. Congestion penalty computation at each edge. Level of Service (LOS) metrics extracted for scoring.

**D – Drainage Proxy Simulation**  
D8 flow accumulation algorithm used to simulate surface water runoff. Flood-risk penalty zones are identified and scored.

**E – Wind / Ventilation Proxy Simulation**  
Gaussian wake propagation model used to approximate ventilation corridors between buildings. Ventilation scores reflect airflow quality through the layout.

**F – KPI Scoring and Ranking**  
All simulation outputs are normalized into dimensionless KPI values. A weighted aggregation function produces a composite score (0–100). Layouts are ranked and the top-K are surfaced for comparison.

**G – Benchmarking**  
Performance benchmarking across boundary shapes, scales, run times, connectivity success rates, and score distributions.

# 02 — HARBOR

### Hierarchical Analysis for Reference-Guided Biodiversity Organism Recognition

**Domain:** AI-Driven eDNA Taxonomy Inference | Bioinformatics  
**Status:** Completed

---

### Overview

HARBOR is an open-set eDNA taxonomy inference system built for large-scale biodiversity recognition from environmental DNA sequences. Traditional taxonomy systems force every sequence into an existing classification – even when the sequence belongs to an unknown organism. HARBOR solves this by combining fast deterministic similarity checks, deep CNN-based sequence embeddings, and statistical novelty detection to produce three honest output types: high-confidence known assignments, explicit NOVEL flags, and explainability metrics. It is designed to scale to multi-marker reference databases (~2.8GB) in alignment-free mode without sacrificing scientific defensibility.

---

### Key Features

**A – Open-Set / Novelty-Aware Classification**  
Distinguishes known sequences from out-of-distribution ones. Novel sequences receive explicit NOVEL flags rather than forced incorrect taxonomy assignments.

**B – Multi-Stage Hybrid Inference (Deterministic → Neural → Statistical)**  
Three-stage pipeline:  
- Stage 1: Deterministic similarity evaluation (fast, fully explainable)  
- Stage 2: Embedding-based ANN similarity search  
- Stage 3: Statistical novelty detection for ambiguous cases  

**C – Marker-Aware Representation Learning**  
Learns biologically meaningful representations per genetic marker, preventing cross-marker interference during embedding and inference.

**D – High-Throughput Similarity Retrieval at Scale**  
Reusable FAISS-based similarity indices are built from reference embeddings, enabling low-latency nearest-neighbor search over millions of sequences.

**E – Scientific Traceability + MLOps Controls**  
Full dataset versioning, experiment tracking, artifact management, and reproducibility controls via DVC and Weights & Biases.

**F – Risk-Controlled System Engineering**  
Designed to mitigate: reproducibility loss, dependency drift, data corruption, model–index mismatch, and silent accuracy degradation.

### Algorithm Stack

**A – Deterministic Feature Engineering (Alignment-Free)**  
Sliding-window decomposition into fixed-length subsequences (k-mer style). Invalid/ambiguous nucleotides are filtered. Hash-based encoding converts sequences into fixed numeric space for deterministic similarity checks and consistent model input.

**B – Deterministic Similarity Evaluation (Fast Known-Match Resolution)**  
Exact and near-exact pattern matching against deterministic reference fingerprints. High-confidence known sequences exit early via this path – maximizing compute efficiency and explainability.

**C – Deep Representation Learning (Marker-Specific CNN Embeddings)**  
Architecture per marker: Embedding layer → 1D Convolutions → Global pooling → Projection + normalization → cosine-distance-ready vectors.

**D – Training Strategy (Open-Set Friendly)**  
Unsupervised instance-level contrastive learning. No hard dependency on taxonomy labels, which improves robustness in open-set inference.

**E – Similarity Index Construction (Offline)**  
Reference embeddings are indexed into per-marker FAISS indices. Index rebuild is decoupled from model training to support scalable updates.

**F – Runtime ANN Retrieval**  
Top-K nearest neighbor search in embedding space provides low-latency similarity evidence for classification decisions.

**G – Statistical Novelty Detection (Open-World Decision Gate)**  
Distance-based and density-based novelty checks applied when confidence thresholds are not met. Explicit NOVEL flags prevent unsafe taxonomy assignments.

# 03 — S.C.A.L.E.

## Structural Causal Analysis of Labor & Education

**Domain:** Causal Modeling | System Dynamics | Policy Evaluation | Computational Economics

**Status:** Published in IEEE Access (January 2026)

**Publication:** _Unbalanced Expansion of Engineering Education in India: A Data-Driven Policy Analysis_

**DOI:** [https://doi.org/10.1109/ACCESS.2026.3704923](https://doi.org/10.1109/ACCESS.2026.3704923)

---

## Overview

S.C.A.L.E. (Structural Causal Analysis of Labor & Education) is an integrated computational policy-analysis framework developed to study the long-term consequences of engineering education expansion in India.

Rather than treating higher education, labor markets, and economic growth as independent systems, S.C.A.L.E. models them as a single dynamic ecosystem connected through delayed feedback loops, causal dependencies, and heterogeneous agent behavior.

The framework combines **System Dynamics (SD)**, **Structural Causal Models (SCM)**, and **Agent-Based Modeling (ABM)** to evaluate how policy interventions influence graduate employability, underemployment, wage dynamics, migration, human-capital retention, and economic productivity over long time horizons.

Unlike traditional forecasting models, S.C.A.L.E. is designed for **counterfactual policy evaluation**, enabling governments and educational planners to test alternative policy strategies before implementation. The published study demonstrates that policies prioritizing educational quality consistently outperform policies focused solely on regulating engineering seat capacity. ([ResearchGate](https://www.researchgate.net/publication/407236404_Unbalanced_Expansion_of_Engineering_Education_in_India_A_Data-Driven_Policy_Analysis?utm_source=chatgpt.com "(PDF) Unbalanced Expansion of Engineering Education in India: A Data-Driven Policy Analysis"))

## Structural Causal Model

Policy variables are represented within a Directed Acyclic Graph (DAG).

[  
Y_{t+k}(do(SeatCapacity=s))

# 04 — IHORMS / IHORMS-X

### Integrated Hospital Operations & Resource Management System

**Domain:** Enterprise Healthcare Platform | Full-Stack Web Application  
**Status:** Completed

---

### Overview

IHORMS is a multi-tenant, role-based hospital operations platform built to unify clinical workflows, resource tracking, billing, insurance, pharmacy, and audit compliance across multiple organizations and branches. It solves the hospital silos problem by providing centralized organization-level oversight alongside branch-level operational autonomy, with strict data isolation between tenants. The system covers the full patient journey: registration → appointment → consultation → telemetry → prescription → billing → insurance → audit. The backend is FastAPI with JWT authentication, and IHORMS-X is the production-ready React frontend that operates exclusively on real backend data with no demo fallbacks.

---

### Key Features

**A – Multi-Tenant Architecture (Org + Branch Isolation)**  
Supports multiple hospital organizations, each with multiple branches. Strict tenant boundaries are enforced via org_id / branch_id scoping throughout the system.

**B – Role-Based Access Control (RBAC) – 8 Roles**  
Roles: `super_admin`, `org_admin`, `branch_admin`, `doctor`, `nurse`, `receptionist`, `pharmacy_staff`, `patient`. RBAC is enforced at both the routing/UI level (ProtectedRoute + redirect) and the API authorization level (JWT claims).

**C – Appointment + Clinical Workflow Management**  
Full workflows for doctor (view/accept/reject/complete appointments, notes, diagnosis, prescription, schedule), receptionist (create/update patients and appointments, availability checks), and patient (history, appointments, prescriptions, doctor browsing).

**D – Telemetry + ICU Monitoring (Nurse Module)**  
Telemetry creation, ICU monitor, alert endpoint, and room/equipment visibility.

**E – Pharmacy Inventory + Order Fulfillment**  
Inventory management, restock, stock adjustment, analytics, low-stock monitoring, and order fulfillment.

**F – Billing + Insurance Workflows**  
Billing creation, bill details, payment capture, insurance claim submission, and claim status updates.

**G – Audit and Compliance Layer**  
Audit logs by entity type, patient access logs, and system events (super admin only).

**H – Production Frontend (IHORMS-X)**  
Sidebar + topbar layout, role-based dynamic navigation, loading/error/empty state handling on every endpoint, backend-down error banner. Strict no-demo-data policy.

## Part A – Pathfinding Research & Benchmarking Framework (35+ Algorithms)

### Key Features

- Benchmarks 35+ algorithms across three categories: classical, metaheuristics, field-based
- Four scenario types: static, fog-of-war, dynamic obstacles, hybrid (dynamic + fog-of-war)
- Standardized testbed: 100×100 grid, 40% blocked, 30% weighted, 30% free tiles
- Metrics: execution time, memory usage, success rate, nodes expanded
- Actionable insight: not just "who wins" but "who wins where"

### Algorithm Coverage

**Classical / Deterministic:** Dijkstra, A*, UCS, Bidirectional A*, Greedy Best-First, BFS, DFS, Theta*  
**Dynamic / Incremental:** D*, D* Lite, LPA*, ARA*, GAA*  
**Anytime / Real-Time:** Anytime A*, RTA*, LRTA*  
**Geometric / Any-Angle:** Field D*, Lazy Theta*, Anya, Angle Propagation A*  
**Crowd / Multi-Agent:** Flow Field  
**Metaheuristics / Nature-Inspired:** PSO, GA, ACO, Max-Min Ant System, Harmony Search, Cuckoo Search, Bat Algorithm, Grey Wolf Optimizer (GWO), Simulated Annealing, Firefly Algorithm

### Algorithm Mathematics

- **A\*:** `f(n) = g(n) + h(n)`, admissible if `h(n) ≤ h*(n)`. Grid heuristic: `h(n) = |x_n - x_g| + |y_n - y_g|`  
- **Weighted A\*:** `f(n) = g(n) + w·h(n), w > 1` – faster, less optimal.  
- **Theta\* (Any-Angle):** Line of Sight shortcuts. `g(neighbor) = g(parent(current)) + d(parent(current), neighbor)`  
- **D\* Lite (Incremental):** `rhs(s) = min_{s' ∈ Succ(s)}(c(s,s') + g(s'))`. Consistent when `g(s) = rhs(s)`.  
- **Flow Field (Swarm):** Global cost-to-go map via BFS/Dijkstra, then vector directions per cell. Scales to RTS crowds.  
- **ACO:** Transition probability: `P_ij = (τ_ij^α · η_ij^β) / Σ(τ_ik^α · η_ik^β)`, pheromone update: `τ_ij ← (1-ρ)τ_ij + Δτ_ij`  
- **Genetic Algorithm:** Fitness = `1 / path_length`. Evolves via selection + crossover + mutation.  
- **Simulated Annealing:** Accept worse with `P(ΔE) = e^(-ΔE/T)`, cool `T → 0`.  
- **Firefly Algorithm:** Attraction decays with distance: `β = β₀ · e^(-γr²)`

### Benchmark Results

| Rank | Algorithm   | Notable Metric          |
|------|-------------|-------------------------|

| #2   | ACO         | High reliability        |
| #3   | MMAS        | Pheromone-bounded stability |

**Dynamic Obstacles (100×100, 3 maps):** Flow Field, ACO, MMAS – 100% success rate.  
**Fog-of-War + Dynamic (50×50, 3 maps):** Most classical algorithms 0% success; Bidirectional A* 33.3%. Validates need for incremental/robust planners.

### Overview

V.E.C.T.O.R is a real-time fraud detection engine that personalizes anomaly detection to each user's behavior profile. Instead of applying a single global fraud model to all users, it continuously learns individual behavior, clusters users into behavioral personas, assigns persona-specific anomaly models, and produces fraud risk scores in under 100ms. The system handles cold-start users via an XGBoost fallback, manages ambiguous transactions through 2FA verification, and provides a live monitoring dashboard. It is designed as a production-grade multi-layer streaming pipeline.

---

### Key Features

**A – Real-Time Transaction Monitoring**  
Stream-based scoring <100ms per transaction.

**B – Dynamic User Profiling**  
Continuously updated behavior profiles: velocity, average spend, frequency, session behavior, time gaps, loan activity, channel usage.

**C – Behavioral Clustering (Persona Engine)**  
Users grouped into clusters (low-spend/high-frequency, high-spend/low-frequency, loan-heavy, abnormal velocity). Fraud baselines differ per cluster.

Each cluster has its own Isolation Forest trained on that cluster's transaction distribution.

**E – Cold-Start Handling**  
XGBoost fallback for users with <10 transactions.

**F – Decision and Verification**  
Fraud score thresholds: <0.4 Approve, 0.4–0.8 2FA, >0.8 Block/Manual review.

Logs suspicious but verified transactions to build trust-vs-suspicion history.

React frontend with real-time charts, Cloudflare Tunnel hosting.

# 18 — DEEP (HARBOR V2)

### Distributed eDNA Evaluation Platform  
**aka HARBOR V2 – Cloud‑Native Bioinformatics Intelligence**

**Domain:** Cloud‑Native Bioinformatics | Multi‑Cloud Serverless AI  
**Status:** In Progress (Operational Prototype)

---

### Overview

DEEP is the fully cloud‑native evolution of the HARBOR eDNA taxonomy inference system. While HARBOR (v1) ran as a local server‑side pipeline, DEEP re‑architects the entire platform for global availability, auto‑scaling, and zero‑ops maintenance by distributing every component across multiple cloud providers. It retains the same scientific core—open‑set, novelty‑aware taxonomy classification from environmental DNA sequences—but delivers it via a serverless, event‑driven architecture that can handle burst workloads from research labs worldwide without manual infrastructure management.

The entire stack is multi‑cloud: Hugging Face hosts the ML inference backend, AWS provides serverless compute and static content delivery, and Cloudflare ties everything together with a custom domain and DNS layer.

---

### Key Features

**A – Serverless Inference Backend**  
Deep learning models (marker‑specific CNN embeddings) are packaged and deployed on **Hugging Face Inference Endpoints**. This removes all GPU/pytorch management overhead and provides auto‑scaling API access for sequence embedding and classification.

**B – Global UI Delivery via CDN**  
The interactive dashboard (terminal‑styled HARBOR UI) is built as a static React application and served globally through **AWS CloudFront**, ensuring low‑latency access from any continent.

**C – API Gateway + Lambda Business Logic**  
All routing, authentication, request validation, and orchestration logic is handled by **AWS Lambda** functions behind **Amazon API Gateway**. This decouples business logic from model inference, allowing independent scaling and updates.

**D – Reliable Asynchronous Job Queuing**  
Long‑running batch classification requests (multi‑sequence FASTA files) are queued through **Amazon SQS**. Lambda workers pull jobs, call Hugging Face for inference, aggregate results, and notify clients. No request is lost during traffic spikes.

**E – Custom Domain & Global DNS**  
**Cloudflare** maps the entire architecture to a memorable custom domain, provides DDoS protection, and routes traffic intelligently between the static UI (CloudFront) and the API Gateway endpoints.

**F – Open‑Set Taxonomy Engine (Core from HARBOR)**  
Preserves all scientific capabilities of HARBOR: alignment‑free feature engineering, marker‑specific CNN embeddings, FAISS‑based ANN retrieval, and statistical novelty detection with explicit NOVEL flags.

**G – MLOps Pipeline on Hugging Face**  
Model retraining, versioning, and deployment are managed through Hugging Face’s model hub, enabling continuous improvement without touching the serverless infrastructure.
